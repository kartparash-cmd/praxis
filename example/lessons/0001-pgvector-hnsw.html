pgvector: tuning an HNSW index

The gap in your repo: pgvector is installed and you query embeddings with <-> (search.py:41), but there's no ANN index — so every search is a sequential scan.

The one idea

pgvector can answer nearest-neighbour queries two ways. Without an index it does an exact scan: compare the query vector to every row. Correct, but O(n). An HNSW index (Hierarchical Navigable Small World) trades a little recall for a huge speedup — it walks a graph of vectors instead of scanning all of them.

CREATE INDEX ON items USING hnsw (embedding vector_cosine_ops);

Two knobs set the recall/speed tradeoff: m and ef_construction at build time, and hnsw.ef_search at query time. Higher ef_search = better recall, slower query. Source: pgvector README → Indexing (primary).

Why not just always index?

HNSW indexes are expensive to build and take memory. Below a few thousand rows a sequential scan is genuinely faster and simpler. The skill matters at scale — which is exactly where this repo is heading.

Recall

Why is a query without an ANN index O(n)? It compares the query vector against every row's vector — an exact scan — so cost grows linearly with row count.
Which query-time setting trades recall for latency, and which way? hnsw.ef_search. Higher = better recall but slower; lower = faster but may miss true neighbours.
When is having NO index the right call? At small row counts (a few thousand), where an exact scan is fast and an HNSW index just adds build cost and memory for no real latency win.
